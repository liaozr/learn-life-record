前端开发规范

一、一般规范

1、命名规范

变量

（1）以英文命名，避免使用中文拼音尽量简易并要求语义化。

（2）全部采用小写方式，如需要分割以-连接。如：my-project-name

（3）前缀应该是名词

（4）尽量在变量名字中体现所属类型，如:length、count等表示数字类型；而包含name、title表示为字符串类型。

// 好的命名方式
let maxCount = 10;
let tableTitle = 'LoginTable';

// 不好的命名方式
let setCount = 10;
let getTitle = 'LoginTable';

常量

  • 全部使用大写+下划线

函数

  • 命名方法：小驼峰式命名法。
  • 命名规范：前缀应当为动词。
  • 命名建议：可使用常见动词约定
// 是否可阅读
function canRead() { return true;
}

// 获取名称
function getName() { return this.name;
}

类名

  • 命名方法：大驼峰式命名法，首字母大写。
  • 命名规范：前缀为名称
class Student{
}
  • 建议尽量使用let而不是用var声明变量。

变量声明方式

2、代码注释

（1）关键步骤必须写注释。

（2）不要写你的代码都干了些什么，而要写你的代码为什么要这么写，背后的考量是什么。当然也可以加入所思考问题或是解决方案的链接地址。

3、代码检查

使用esLint对javascript 代码进行检查。

二、HTML规范

1、文档类型

使用HTML5文档类型声明。

<!DOCTYPE html>

2、结构(HTML)、表现(CSS)、行为(JS)代码分离。

  • HTML文件中只包含标签，将表现放入样式表中，将所有动作行为放入脚本文件中。在HTML的中引入样式文件，在之前引入脚本文件。

3、meat声明

<meta name="viewport" content="width=device-width, initial-scale=1.0"/>

让网页宽度适应屏幕宽度，多用于移动端。

<meta http-equiv="X-UA-Compatible"content="IE=edge,chrome=1" />

<meta name="renderer" content="webkit|ie-comp|ie-stand">

让360浏览器默认进入极速模式

4、img标签必须添加alt属性。

  • 在图像无法显示时的替代文本。在网速慢、图片src属性错误、浏览器禁用图片或是屏幕阅读器中alt属性很重要，且利于SEO。

5、标签规范

html标签属性必须小写
标签、属性、属性命名由小写英文、数字和_组成，且所有标签必须闭合，属性值必须用双引号""
避免冗余标签
段落文字应该用<p>而不应该使用<br>换行
列表项放<ul>,<ol>,<dl>，不要使用<div>或者<p>
<input>应该使用for属性绑定<label>
form button应该指定type类型，<button type="submit">xxx</button>

三、CSS、SASS规范

  • 推荐语义化的名称
  • 避免使用ID作为样式名称
  • 避免使用标签名作为样式名称 省略"0"值后面的单位(px,rem,em)
  • 颜色表示推荐使用十六进制表示法，除非一些特殊需要每个声明必须以";"结尾，除非使用sass等语法
  • css属性选择器或者属性值必须使用"双引号，但url(path)不要使用引号推荐使用css缩写属性，比如 font、background等

1、css选择器层级

错误示范：

.content_box form ul li {height:30px;margin:10px;}

错误原因：样式指向不精准，使得受影响的标签范围太大，当多个页面同时引用同一样式表时，容易出现不可预见的问题。

#final #content .con_border .system_time .default_checkbox {}

错误原因：选择器太多，权重太高，不利于后期维护。

.access_control > ul > li > ul > li > input{
    margin-right:10px;
}

错误原因：选择器太多，影响运行效率，不利于样式复用。尽量避免使用后代选择器。
建议：层级尽量不超过三级，防止出现权重太高，不利于后期的覆盖与维护。

2、!important

尽量避免使用!important，因为这是css里面的终极大招，用多了伤元气。

3、合理的避免使用ID选择器。

一般情况下ID不应被应用于样式表中。因为ID样式不能被复用且每个页面中只能使用一次ID。还有一个原因是ID选择器权重很高。

4、尽量避免使用元素选择器

错误示范：

form ul li em{

}

推荐写法：

.content .title{

}

5、文件命名、ID、CLASS等所有命名避免以下词汇。

一些浏览器会将含有这些词的作为广告拦截，文件命名、ID、CLASS等所有命名避免以上词汇。 ad、ads、adv、banner、sponsor、gg、guangg、guanggao等。

6、命名风格

使用连字符"-"来作为单词的分隔符，避免使用"_"下划线

/*不推荐*/
.demoimage {}
.error_status {}
/*推荐*/
.ads-sample {}

7、声明顺序

  • 位置属性(position, top, right, z-index, display, float等)
  • 大小(width, height, padding, margin)
  • 文字系列(font, line-height, letter-spacing, color- text-align等)
  • 背景(background, border等)
  • 其他(animation, transition等)
.box {
    display: block; 
    position: absolute; 
    left: 30%;
    right: 30%; 
    overflow: hidden; 
    margin: 1em; 
    padding: 1em;
    background-color: #eee; 
    border: 3px solid #ddd;
    font-family: 'Arial', 
    sans-serif; 
    font-size: 1.5rem;
    text-transform: uppercase;
}

8、嵌套顺序

1.当前选择器的样式属性
2.当前选择器的伪类样式(:first-letter,:hover,:active etc)
3.当前选择器的伪类元素(:before,:after)
4.子选择器部分

.parent{
    color:red; 
    background:url(img.png); 
    &:first-letter{
        font-size:30px; 
        font-weight:bold;
    }
    &:before{
        position:absolute;
    }
    .child{
        width:100px;
    }
}

四、JAVASCRIPT规范

1、等同== 和严格等===的区别

==， 两边值类型不同的时候，要先进行类型转换，再比较。===，不做类型转换，类型不同的一定不等。

使用 === 精确的比较操作符，避免在判断的过程中，由 JavaScript 的强制类型转换所造成的困扰。例如

(function(log){
  'use strict';

  log('0' == 0); // true
  log('' == false); // true
  log('1' == true); // true
  log(null == undefined); // true

  var x = {
    valueOf: function() {
      return 'X';
    }
  };

  log(x == 'X');

}(window.console.log));

总是使用 === 精确的比较操作符，避免在判断的过程中，由 JavaScript 的强制类型转换所造成的困扰。

如果你使用 === 操作符，那比较的双方必须是同一类型为前提的条件下才会有效。 在只使用 == 的情况下，JavaScript 所带来的强制类型转换使得判断结果跟踪变得复杂。

2、二元和三元操作符

  • 操作符始终写在前一行, 以免分号的隐式插入产生预想不到的问题
let x = a ? b : c;
let y = a ?
    longButSimpleOperandB : longButSimpleOperandC;
let z = a ?
        moreComplicatedB :
        moreComplicatedC;

3、条件(三元)操作符 ( ? : )

  • 三元操作符用于替代 if 条件判断语句。
if (val != 0) {
  return foo();
} else {
  return bar();
}
// Recommended
return val ? foo() : bar();

4、字符串

  • 字符串统一使用' '单引号，避免与html中的"冲突
  • 需要拼接，尽量少用+，多使用模板字符串
let str1 = 'yang'

let string = `hello ${str1}`

5、协议

引入外部资源时，url所指向的具体路径不要指定协议部分，防止资源协议无法确定


<!--不推荐-->
<script src="http://cdn.com/foundation.min.js"></script>
<!--推荐-->
<script src="//cdn.com/foundation.min.js"></script>

6、代码注释风格

  • 对于不能自解释的代码，function必须使用JSDoc风格的注释
/**
*method description
*@param {type} pramaA description
*@param {type} pramaB description
*@returns {type} description
*/
function method (pramaA, pramaB) { return true;
}

五、Vue规范

1、组件/实例的选项顺序

  • name
  • components / directives / filters
  • extends / mixins
  • model / props / propsData
  • data / computed
  • watch / 生命周期钩子
  • Methods

2、元素特性的顺序

  • is
  • v-for / v-if / v-else-if / v-else / v-show / v-cloak
  • v-pre / v-once
  • id
  • ref / key / slot
  • v-model
  • v-on
  • v-html / v-text

3、事件的控制

  • 不要使用 this.parent / this.root 进行控制, 会导致流程难以跟踪. 可使用事件中心进行代替.
  • 事件中心每添加事件, 需要注明事件的 名称 / 来源文件 / 流向文件, 以及事件的作用.
  • 事件移除时需要移除事件中心对应事件注明.
  • 组件当中有原生绑定事件, 注意在组建注销前进行移除

六、版本控制

说明：本公司使用Gitlab作为版本控制。

1、分支管理（与后端一致）

本公司的每个项目在分支管理方面，必须统一采用以下工作流程。

（1）主干分支

项目初始化时会自动创建，是受到保护的分支，名为master。此分支的代码必须始终保持与正式环境一致，每次预发布正式版本测试通过之后，必须基于此分支新建标签，标签名以“<版本号>.RELEASE”格式进行命名。

（2）测试分支

项目初始化完成之后即可创建（基于master），是长期性分支（一般情况下不会被删除），直接命名为test。此分支的代码必须始终保持与测试环境一致。

（3）特性分支

每次开发新功能或修补BUG时创建，是短期性分支（发布后可以删除），若是开发新功能的分支，则以“feature/<版本号>”格式进行命名；若是修补BUG的分支，则以“hotfix/<版本号>”的格式进行命名。通常基于主干分支或其他特性分支（可以基于多个）创建，具体得根据实际情况而定，一般情况下，若同时满足以下条件，则可以考虑基于其他特性分支创建。

  • 本次需要创建的特性分支与其他未发布的特性分支有很大的关联，通常集中相同的类、页码或模块中开发；
  • 被基于创建的其他特性分支的需求被废弃、搁置的可能性不大，且不会存在重大BUG。

需要明确的是，以上基于其他特性分支创建只是一种建议，需要开发人员根据实际情况进行权衡利弊。但是当本次需要创建的特性分支与其他未发布的特性分支存在很大关联时，不管基于哪个分支创建，都会存在一些弊端。

基于主干分支创建存在的弊端：

  • 本次需要创建的特性分支与有很大关联的其他未发布特性分支合并时，必定会存在很多冲突需要解决；
  • 在本次需要创建的特性分支中开发过程中，可能无法复用有很大关联的其他未发布特性分支的通用代码。

基于其他有很大关联的特性分支创建存在的弊端：

  • 若其他有很大关联的特性分支的需求被废弃或搁置，则本次需要创建的分支在发布时会连带其他特性分支的代码一同发布，这就显得有些多余，甚至会受到它的影响；
  • 若其他有很大关联的特性分支的代码存在重大BUG，则本次需要创建的分支在发布时会连带此重大BUG一同发布，也会受到它的影响。

（4）发布分支

发布正式版本时创建（基于master），是短期性分支（发布完成后可删除）。通常以“release/<版本号>”格式进行命名。此分支的代码是预发布代码，可以将本次需要发布的所有特性分支依次合并到此分支，若预发布后存在BUG，则可直接在此分支进行修补。预发布完成之后，必须将此分支合并到master分支。

2、代码质量

一切代码必须同时满足以下两个条件方可提交到Gitlab。

  • 本次提交的代码已经符合本规范， 提交前执行npm run lint检测代码规范；
  • 本次提交的代码已经通过测试，至少是启动不报错，否则会影响他人的开发进度。
